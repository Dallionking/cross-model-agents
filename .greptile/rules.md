# Code Review Rules — Sigma Protocol

## Security

Every user-facing endpoint must verify authentication. Never trust client-supplied IDs without server-side ownership verification. All webhook endpoints must verify signatures before processing payloads.

Never interpolate user input into SQL, shell commands, or LLM prompts via template literals. Use parameterized queries, structured message arrays, and validated inputs.

Secrets belong in environment variables or secret managers. Never commit API keys, tokens, or passwords. Check for common key prefixes: `sk-`, `ghp_`, `gho_`, `AKIA`, and patterns like `password =` or `api_key =`.

## Anti-Slop Patterns

Code generated by AI models often contains these artifacts. Flag them:

- Comments that restate the code: `// This function returns the user` above `function getUser()`
- Separator comments: `// ---`, `// ===`, `// ****`
- Hedging in error messages: "might", "could perhaps", "it seems like"
- Filler words in variable names or comments: "robust", "seamless", "leverage"
- Unnecessary type annotations that TypeScript can infer
- Empty catch blocks or catch blocks that only log and swallow errors

Good comments explain WHY a decision was made, not WHAT the code does.

## Error Handling

Handle errors at system boundaries (API calls, database queries, file I/O, external services). Internal function calls between modules generally don't need defensive error handling — trust internal code.

Error messages must say WHAT went wrong AND what to do about it. Not just "An error occurred" — include the context: "Failed to fetch user profile: auth token expired. Re-authenticate and retry."

Never swallow errors silently. At minimum, log with context and re-throw.

## TypeScript

- Use `unknown` instead of `any`. Narrow with type guards.
- Use named exports, not default exports.
- Don't create barrel exports (`index.ts` re-exporting everything).
- Prefer `const` assertions and `satisfies` over type casting.
- Let TypeScript infer return types for simple functions. Only annotate when the inference is wrong or the function is part of a public API.

## Testing

- Test files should test behavior, not implementation details.
- Don't mock what you don't own — wrap third-party APIs in adapters, mock the adapter.
- Every async test path needs both success and failure cases.
- No `test.skip` without a linked issue explaining why.

## React / Frontend

- Server Components by default. Only add `"use client"` when you need browser APIs, event handlers, or hooks.
- Query by role first in tests: `getByRole("button", { name: /submit/i })`.
- Don't duplicate state that can be derived from existing state or props.
- Validate all user inputs with Zod or equivalent before submission.

## Performance

- Never do full-table scans on collections expected to exceed 1K documents. Use indexes.
- Any list that may exceed 50 items must use pagination.
- Don't block mutations on external API calls — schedule async actions instead.
- Lazy-load heavy components and routes.
